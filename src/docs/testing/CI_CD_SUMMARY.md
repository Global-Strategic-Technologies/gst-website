# CI/CD Pipeline Summary

## Pipeline Overview

**Workflow:** `.github/workflows/test.yml`
**Triggers:** Push to `master` or `dev`, PRs to `master`
**Deduplication:** Concurrency group cancels redundant runs when both push and PR fire for the same branch
**Path filtering:** Skips docs-only changes (`**.md`, `src/docs/**`, `.claude/**`)

## Jobs

### Unit & Integration Tests
- Runs on Node 22.x
- Executes `npm run test:coverage`
- Coverage summary (statements, branches, functions, lines) posted to GitHub Actions step summary
- **Timeout:** 10 minutes

### E2E Tests (Playwright)
- Depends on unit tests passing
- Builds the application (`npm run build`) and verifies output
- Runs `npm run test:e2e` across chromium, firefox, webkit
- Playwright browsers cached between runs
- Playwright HTML report uploaded as artifact (7-day retention)
- **Timeout:** 20 minutes

## Concurrency & Deduplication

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.head_ref || github.ref_name }}
  cancel-in-progress: true
```

A new push to the same branch cancels any in-progress run, preventing wasted compute.

When a PR is open and you push to `dev`, both `push` and `pull_request` events fire. Both resolve to the same concurrency group (`Test Suite-dev`), so the earlier run is cancelled вҖ” only one completes.

| Scenario | Events fired | Runs to completion |
|---|---|---|
| Push to `dev`, no PR open | `push` | 1 |
| Push to `dev`, PR open | `push` + `pull_request` | 1 (earlier cancelled) |
| Merge to `master` | `push` | 1 |

## Coverage

Coverage is generated by `vitest run --coverage` (v8 provider) and reported as a markdown table in the GitHub Actions step summary. No external service required.

**Target:** 70%+ line coverage (see CLAUDE.md)

## Vercel Integration

Tests and deployment are independent:

- **GitHub Actions** runs tests (feedback)
- **Vercel** builds and deploys (delivery)
- Both trigger simultaneously on push
- Branch protection ensures PRs pass tests before merge

Vercel is not involved in coverage or test reporting.

## Branch Protection

PRs to `master` require:
- All test status checks passing
- At least 1 approval
- Up-to-date branch

### Required Status Checks
- Test Suite / Unit & Integration Tests
- Test Suite / E2E Tests (Playwright)
